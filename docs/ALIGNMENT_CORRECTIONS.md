# Architecture Alignment Pass - Corrections Summary

## Overview

This document summarizes the corrective architecture alignment pass performed on `haul-svc-notifications` to eliminate architectural drift and enforce strict boundary compliance.

**Objective**: Fix violations of the canonical architecture. NOT to add features, expand scope, or introduce abstractions.

---

## Mandatory Corrections Applied

### 1. ✅ Pinpoint is the ONLY Outbound Messaging System

**Violations Removed**:
- Removed `SES_FROM_ADDRESS` environment variable
- Removed `SES_SOURCE_ARN` environment variable
- Removed `EmailChannelSESPolicy` IAM policy
- Removed all SES troubleshooting steps from documentation
- Removed SES sandbox check instructions
- Removed references to "direct SES integration"

**Corrections Made**:
- Added `PINPOINT_EMAIL_FROM_ADDRESS` as the single configuration variable
- Updated Pinpoint email channel to use `PINPOINT_EMAIL_FROM_ADDRESS`
- Updated email handler to use `FROM_ADDRESS = process.env.PINPOINT_EMAIL_FROM_ADDRESS`
- Clarified in `resources/ses.yml` that SES is an internal transport layer beneath Pinpoint only

**Result**: SES is now mentioned ONLY as an assumed prerequisite for Pinpoint email delivery, not as a first-class integration point.

---

### 2. ✅ Pinpoint Application ID Provided at Runtime

**Violations Removed**:
- Removed manual CloudFormation query instructions from DEPLOYMENT.md
- Removed all documentation instructing users to retrieve Pinpoint App ID manually

**Corrections Made**:
- Added `PINPOINT_APP_ID` to provider-level environment variables in serverless.yml:
  ```yaml
  PINPOINT_APP_ID:
    Ref: HaulPinpointApplication
  ```
- Removed redundant `PINPOINT_APP_ID` from email handler function-level env vars
- Updated documentation to state "The Pinpoint Application ID is automatically created and injected into all Lambdas via CloudFormation"

**Result**: Pinpoint App ID is now injected at runtime for all Lambdas via CloudFormation reference. No manual steps required.

### 3. ✅ EventBridge Ownership Not Violated

**Violations Removed**:

- Removed all Terraform code examples for EventBridge targets
- Removed all Terraform code examples for Lambda permissions
- Removed instructions to "Add the orchestrator Lambda as a target to EventBridge rules"

**Corrections Made**:

---

### 4. ✅ Notifications Service Does NOT Query User Identity Tables

**Violations Removed**:
- Removed `getUserEmail()` function with User table query stub
- Removed all references to `User-{env}` DynamoDB table
- Removed documentation suggesting "Query User table from DynamoDB"
- Removed assumptions about user profile schemas (e.g., `email` attribute)

**Corrections Made**:
- Created `resolveUserEmail()` utility function with explicit architectural boundary comment:
  ```javascript
  /**
   * ARCHITECTURAL BOUNDARY:
   * The notifications service MUST NOT query User identity tables directly.
   * 
   * TODO: Integrate with identity/profile service or endpoint for contact resolution
   */
  async function resolveUserEmail(userId) {
    // DO NOT query User-{env} table directly
    // DO NOT assume user profile schema
    return null; // Stub
  }
  ```
- Updated documentation to explicitly state this boundary constraint
- Added TODO comments for identity service integration options

**Result**: The service no longer assumes access to User tables. Contact resolution is clearly marked as requiring external integration.

---

### 5. ✅ NotificationInbox Identity Semantics Clarified

**Violations Removed**:
- Fixed incorrect usage of `event_id` as notification identifier in delivery status updates
- Removed ambiguity about which ID is the primary key

**Corrections Made**:
- Added documentation to `writeToInbox()`:
  ```javascript
  /**
   * Write notification record to NotificationInbox
   * notification_id is generated by this service and is the primary identifier
   * event_id is a foreign reference to the originating domain event
   */
  ```
- Fixed `processRecipient()` to return `notification_id` from `writeToInbox()`
- Updated `enqueueToChannel()` to include `notification_id` in SQS messages
- Updated `updateDeliveryStatus()` to use `notification_id` (not `event_id`) for DynamoDB updates
- Added null check for `notification_id` in delivery status updates

**Result**: 
- `notification_id` is service-generated and is the primary identifier for NotificationInbox records
- `event_id` is a foreign reference to the domain event that triggered the notification
- Delivery status updates now use the correct identifier

---

### 6. ✅ Resolver Responsibilities Strictly Enforced

**Corrections Made**:
- Updated `resolvers/base.js` with comprehensive boundary documentation:
  ```javascript
  /**
   * RESOLVER RESPONSIBILITIES (MUST DO):
   * - Accept the validated canonical ingress schema
   * - Return a list of resolved recipient user_ids
   * 
   * RESOLVER BOUNDARIES (MUST NOT DO):
   * - Evaluate notification preferences
   * - Determine delivery channels
   * - Perform delivery logic
   * - Render templates
   * - Write to DynamoDB
   * - Send messages to SQS queues
   * - Call Pinpoint or any external messaging system
   * 
   * Resolvers are LOOKUP ONLY.
   */
  ```
- Added boundary enforcement comments to all 5 resolver implementations
- Updated documentation to reinforce "Resolvers perform LOOKUP ONLY"
- Removed misleading TODO comment from `JobPostedResolver` that mentioned "Respect provider notification preferences" (preferences are orchestrator responsibility)

**Result**: Resolver boundaries are now explicit and enforced via documentation. No ambiguity about resolver scope.

## Conclusion

This corrective pass eliminated all architectural violations and tightened boundaries without adding features or expanding scope.

The service now strictly adheres to the canonical architecture:
- **Single outbound system**: Pinpoint
- **Runtime injection**: Pinpoint App ID via CloudFormation
- **External ownership**: EventBridge ingress managed centrally
- **Boundary enforcement**: No User table queries, resolvers are lookup-only
- **Identity clarity**: notification_id vs event_id semantics fixed
- **Phase discipline**: Push/SMS non-operational until explicitly enabled

All changes preserve V1 simplicity while ensuring architectural correctness.
